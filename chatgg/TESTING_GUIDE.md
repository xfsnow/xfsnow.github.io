# 流式 vs 非流式响应测试指南

## 快速开始

### 1. 打开程序
在浏览器中打开 `chatgg/index.html`

### 2. 配置 API
- 点击左下角"设置"按钮
- 输入你的 API 密钥
- **重要**：勾选或取消"使用流式响应"来切换模式

### 3. 进行测试

#### 测试步骤
1. 打开浏览器控制台（按 F12）
2. 切换到 Console 标签
3. 发送几条消息，观察控制台输出

#### 测试用例建议

**短回复测试：**
```
画一个圆
```

**中等回复测试：**
```
画一个等边三角形，并标注三个角
```

**长回复测试：**
```
证明勾股定理，并画出示意图
```

### 4. 查看性能数据

每次发送消息后，控制台会显示：

```javascript
// 流式响应示例
[性能] 首字延迟: 342 ms
[性能] 流式响应 - 总耗时: 2856 ms, 首字延迟: 342 ms
[性能统计] {streamCount: 1, nonStreamCount: 0, streamTimes: [2856], nonStreamTimes: []}

// 一次性响应示例
[性能] 一次性响应 - 总耗时: 2423 ms
[性能统计] {streamCount: 0, nonStreamCount: 1, streamTimes: [], nonStreamTimes: [2423]}
```

## 对比要点

### 关键指标

| 指标 | 说明 | 如何观察 |
|------|------|---------|
| 首字延迟 | 从发送到看到第一个字的时间 | 流式模式会显示此数据 |
| 总耗时 | 从发送到完整接收的时间 | 两种模式都会显示 |
| 用户体验 | 主观感受 | 自己体会哪种更舒服 |
| Token 消耗 | API 费用 | 查看 API 提供商后台 |

### 测试建议

1. **每种模式至少测试 5 次**，取平均值
2. **使用相同的问题**进行对比
3. **记录以下数据**：
   - 首字延迟（仅流式）
   - 总耗时
   - 回复长度（字符数）
   - 主观体验评分（1-10分）

### 示例测试表格


## 预期结果

### 理论预期

根据一般经验：

**流式响应：**
- 首字延迟：200-500ms
- 总耗时：比一次性慢 10-20%
- 用户体验：⭐⭐⭐⭐⭐

**一次性响应：**
- 首字延迟：2000-5000ms（无中间反馈）
- 总耗时：比流式快 10-20%
- 用户体验：⭐⭐⭐

### 实际影响因素

1. **网络状况**：网络越差，流式优势越明显
2. **回复长度**：回复越长，流式体验越好
3. **API 服务商**：不同服务商的实现可能有差异
4. **模型类型**：不同模型的生成速度不同

## 如何判断哪种更好？

### 选择流式响应，如果：
- ✅ 重视用户体验
- ✅ 回复通常较长（>200字）
- ✅ 网络环境稳定
- ✅ 希望用户看到 AI "思考"过程

### 选择一次性响应，如果：
- ✅ 追求代码简洁
- ✅ 回复通常较短（<100字）
- ✅ 网络环境不稳定
- ✅ 不介意用户多等几秒

## 我的建议

对于 ChatGG 项目（GeoGebra 数学绘图）：

### 推荐使用流式响应

**原因：**
1. 数学解释通常较长，流式体验更好
2. 用户可以边看解释边理解
3. 首字延迟低，感觉响应更快
4. Token 消耗相同，没有额外成本

**优化建议：**
- 保持流式接收文字
- 但等到完整响应后再解析和渲染 GeoGebra 图形
- 这样既保持了良好的文字体验，又避免了部分渲染的问题

## 常见问题

### Q: 为什么两种方式 Token 消耗相同？
A: API 计费基于输入+输出的 token 数量，与传输方式无关。流式只是改变了数据传输的方式，不改变内容。

### Q: 流式响应会更贵吗？
A: 不会！Token 数量完全相同，费用也相同。

### Q: 哪种更快？
A: 
- 首字延迟：流式快 5-10 倍
- 总完成时间：一次性可能快 10-20%
- 感知速度：流式明显更快（因为能看到进度）

### Q: 我应该用哪种？
A: 建议先用流式（默认），如果觉得总速度不够快，再尝试一次性响应对比。
